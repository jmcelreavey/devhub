# Create Pull Request

Merged from OpenCode skill `create-pr` and prior Codex `gh`-focused workflow.

## Modes

- **Draft mode**: Inspect branch state, summarize changes, draft title/body—**no** `git push`, commit, or Jira mutation unless the user asked.
- **Execution mode**: Only when the user clearly asked to create/open/submit the PR—then commit/push/`gh pr create` as needed, and mutate Jira **only** if explicitly requested.

Default to draft mode when the user is ambiguous.

When enough context is available, also provide a Slack-ready review message the user can copy and paste.

## Prerequisites

- `gh` installed and authenticated (`gh auth status`).
- `git` available; working tree may need commits before PR creation.

## Core Rules

1. **Separate analysis from mutation**—summarize before changing remote state.
2. **Do not mutate Jira by default** (comments, transitions) unless the user explicitly asked.
3. **Do not push or commit** unless the user asked for PR creation or a commit.
4. **Use the repo PR template** when present (`.github/pull_request_template*`, etc.).
5. **Infer PR title style** from recent merged PRs and branch names when the repo is Jira-heavy.
6. **Avoid file lists and diff dumps** in the PR body unless the template requires a concise testing section—prefer short prose: problem, fix, how to verify.
7. **No attribution footers**—do not add lines like “Made with Cursor”, “Made with [Cursor](https://cursor.com)”, “Generated by Cursor”, “Generated by AI”, or similar to commit messages, PR titles/bodies, or merge descriptions.
8. **When providing the Slack message**, default to a Slack-safe plain-text format with bare URLs because chat UIs may rewrite Slack mrkdwn during copy.

## Workflow

### 1) Gather context

- Branch, upstream, `git status`, staged/unstaged diff
- Recent commits for message style
- Recent merged PR titles/bodies (`gh pr list --state merged --limit 3`)
- PR template and `CONTRIBUTING` / `AGENTS.md` if present

### 2) Resolve ticket context

- Infer Jira key from branch name, commits, or user message.
- If no key and the repo expects one, ask—or use repo convention for missing ticket if documented (e.g. placeholder only when user approves).
- Never invent ticket titles or descriptions.

### 3) Base branch

- `gh repo view --json defaultBranchRef --jq .defaultBranchRef.name` (or ask if unknown).

### 4) Branch naming (when creating a branch)

- If on base branch or branch name lacks ticket: `<JIRA>-<short-description>` (lowercase hyphens).

### 5) Commit and push (execution mode only)

If working tree has changes and user asked to open PR:

1. `git status -sb`
2. Stage intended files (`git add` as appropriate)
3. Commit: `"<JIRA>: <short-description>"` matching team style
4. `git push -u origin <branch>`

If push fails, explain auth/permissions—do not retry blindly.

### 6) Compose title and body

- Match merged PR punctuation (e.g. `ABC-123: ...` vs `[ABC-123] ...`).
- Body: minimal prose—issue, fix, verification. Link tickets that are in scope only.
- Strip or omit any default “Made with Cursor” (or similar) footer; those must not appear in published PR text.

### 6a) Compose the Slack review message

- Provide a copy-paste-ready Slack message whenever a PR URL exists or is about to exist.
- Present it in a dedicated section labeled exactly `Slack message:` followed by a fenced `text` block so it is easy to copy.
- Default to a Slack-safe plain-text format that survives copy-paste through chat UIs: use bare URLs, not Slack mrkdwn links.
- Use the repo name in inline code formatting as the review target, for example `` `capi-serverless` ``.
- When both links are known, format it like:

  `PR ready for `capi-serverless` - DAD-3948 validate MSN related slideshow posts`
  `PR: PR_URL`
  `Jira: JIRA_URL`

- Replace `ready for` with a short audience-appropriate review prompt only when the user asked for a different phrasing.
- Use the real Jira key and summary when known. Never invent ticket text.
- If there is no Jira ticket, still provide the Slack message with just the PR link and concise description.
- In draft mode, if the PR does not exist yet, provide a template with placeholders such as `PR_URL` and fill in the rest from known context.
- Only provide raw Slack mrkdwn like `<url|text>` as a secondary option when the user explicitly asks for it.

Example output:

Slack message:

```text
PR ready for `capi-serverless` - DAD-3948 validate MSN related slideshow posts
PR: PR_URL
Jira: JIRA_URL
```

### 7) Create PR

Use heredoc for body to avoid shell interpolation of backticks:

```bash
GH_PROMPT_DISABLED=1 GIT_TERMINAL_PROMPT=0 gh pr create --title "$TITLE" --body "$(cat <<'EOF'
<PR_BODY>
EOF
)" --head "$BRANCH" --base "$BASE"
```

Return the PR URL and final title/body.

Also return the Slack-ready message.

## Fallbacks

- No Jira key: follow repo style or ask user.
- No template: short sections—Summary, Changes, Verification.
- No `jc`/MCP: continue without ticket enrichment.
- Non-BI repos: skip Jira-specific behavior.

## Examples

- “Draft a PR description” → draft mode only.
- “Open the PR” / “run gh pr create” → execution mode after confirming changes.
