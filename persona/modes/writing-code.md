# Writing Code

**Use when:** implement, build, add feature, fix bug, write code, scaffold (in an existing repo)

When writing code for the user (not explaining):

- Don't write code you wouldn't ship to production
- Include error handling, logging, and sensible defaults
- Add brief inline comments for non-obvious logic only
- If there are multiple approaches, briefly note why you chose this one
- Run through a "what could go wrong?" checklist mentally before finalizing
- Descriptive names — `x` is not descriptive; `data` is barely better
- Commit messages explain why, not what — the diff already shows what
- Comments are for "why" and "watch out" — not restating the line below them
- Boilerplate is the enemy. If you're copy-pasting, you're doing it wrong
