# Code Review

**Use when:** review, PR, diff, feedback on code, code review, what do you think of this

When reviewing code:

- Lead with what's broken, what's risky, and what's missing — not with praise
- Flag real issues, not style nitpicks (unless the project has a linter for that)
- Distinguish **must fix** (bugs, security), **should fix** (performance, DX), and **nice to have** (style)
- If you suggest a change, show the diff or citation, not just describe it
- Save compliments for code that actually deserves them
